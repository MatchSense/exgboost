# Contributing to EXGBoost

Thanks for your interest in contributing! This guide will help you understand our codebase conventions and safety requirements.

## Quick Setup

### Prerequisites

- Elixir and Erlang
- Make and CMake
- On macOS: `brew install libomp`

### First-time compilation

```bash
export CC_PRECOMPILER_PRECOMPILE_ONLY_LOCAL=true
mix deps.get
mix compile
mix test
```

## NIF Development - DO's and DON'Ts

### Memory Management

#### ✅ DO:

- **Prefer `enif_make_new_binary()` when returning binaries directly to Elixir**
  ```c
  ERL_NIF_TERM binary_term;
  unsigned char *dest = enif_make_new_binary(env, size, &binary_term);

  if (dest == NULL && size != 0) {
    return exg_error(env, "Failed to allocate");
  }

  if (size != 0) {
    memcpy(dest, source, size);
  }

  return exg_ok(env, binary_term);
  ```
  The binary belongs to the NIF environment immediately and is cleaned up by the BEAM when the returned term is no longer reachable.

- **Copy XGBoost output immediately**. XGBoost owns returned pointers, so validate the result and copy the data before returning from the NIF:
  ```c
  const float *xgb_output = NULL;

  int result = XGBoosterPredict(..., &xgb_output);

  if (result != 0) {
    return exg_error(env, XGBGetLastError());
  }

  ERL_NIF_TERM binary_term;
  unsigned char *dest = enif_make_new_binary(env, size, &binary_term);

  if (dest == NULL && size != 0) {
    return exg_error(env, "Failed to allocate");
  }

  if (size != 0) {
    memcpy(dest, xgb_output, size);
  }

  return exg_ok(env, binary_term);
  ```

- **Free temporary C buffers** with the matching allocator:
  ```c
  char *temp = enif_alloc(size);
  // ... use it ...
  enif_free(temp);
  ```

  Allocator pairs:

  - `enif_alloc()` → `enif_free()`
  - `malloc()` → `free()`
  - `enif_alloc_binary()` → `enif_release_binary()`, unless ownership is transferred with `enif_make_binary()`
  - `enif_make_new_binary()` → no manual release
  - `enif_alloc_resource()` → `enif_release_resource()` after creating the resource term

- **Use a single cleanup path** for functions with multiple allocations:
  ```c
  char *first = NULL;
  char *second = NULL;
  ERL_NIF_TERM ret;

  // ... allocate and use buffers ...

  END:
    if (first != NULL) {
      enif_free(first);
    }

    if (second != NULL) {
      enif_free(second);
    }

    return ret;
  ```

- **Initialize all pointers to `NULL`** before any operation that can fail.

- **Check every allocation** before using the returned pointer.

- **Clear local ownership after transfer**. For example, after converting an `ErlNifBinary` to a BEAM term with `enif_make_binary()`, do not release it manually.

#### ❌ DON'T:

- **Don't expose or persist native memory addresses across NIF calls**.

  Pointer-bearing Array Interface JSON may exist temporarily inside a NIF, but it must be constructed and consumed before that NIF invocation returns.

- **Don't return XGBoost-owned pointers to Elixir**. Copy their contents into BEAM-owned terms first.

- **Don't assume pointers remain valid across NIF calls**.

- **Don't dereference an address supplied by Elixir**, even when it is represented as a valid integer.

- **Don't free XGBoost-owned output pointers**:
  ```c
  const float *output = NULL;
  XGBoosterPredict(..., &output);

  // DON'T:
  // free(output);
  // enif_free(output);
  ```

- **Don't call `enif_release_binary()` after `enif_make_binary()`**. Ownership has already transferred to the returned term.

- **Don't use `enif_alloc_binary()` by default for data returned directly to Elixir**. It is valid, but `enif_make_new_binary()` is simpler when no separate ownership period is required.

### Pointer and Array Interface Safety

#### ✅ DO:

- **Derive binary addresses inside the NIF that consumes them**:
  ```c
  ErlNifBinary data_bin;

  if (!enif_inspect_binary(env, binary_term, &data_bin)) {
    return exg_error(env, "Binary argument required");
  }

  uintptr_t address = (uintptr_t)data_bin.data;

  // Build standards-compliant Array Interface JSON using `address`
  // and call XGBoost before this NIF returns.
  ```

- **Pass backing binaries and metadata into the NIF**, rather than passing an address generated by an earlier NIF call.

- **Keep Array Interface JSON standards-compliant**. Extra ownership information belongs in the EXGBoost NIF API, not in the JSON passed to XGBoost.

- **Parse and copy XGBoost-returned Array Interfaces in the same NIF invocation** that receives them. Return copied binaries and metadata to Elixir, not reusable addresses.

#### ❌ DON'T:

- **Don't use a two-NIF pointer flow**:
  ```elixir
  address = EXGBoost.NIF.get_binary_address(binary)
  EXGBoost.NIF.predict(json_containing_address)
  ```

- **Don't expose a general-purpose `get_binary_from_address(address, size)` NIF**. It is an arbitrary memory-read primitive and can crash the entire BEAM or expose unrelated memory.

### Type Safety

#### ✅ DO:

- **Use `uintptr_t` for pointers**:
  ```c
  uintptr_t address = (uintptr_t)ptr;
  ```

- **Use `PRIuPTR` when formatting pointer-sized integers**:
  ```c
  snprintf(buffer, buffer_size, "%" PRIuPTR, address);
  ```

- **Use `ErlNifUInt64` at the BEAM integer boundary**:
  ```c
  ErlNifUInt64 value;

  if (!enif_get_uint64(env, term, &value)) {
    return exg_error(env, "Expected a non-negative integer");
  }
  ```

- **Use `bst_ulong` for XGBoost counts and dimensions**. XGBoost defines it as a 64-bit unsigned type.

- **Convert between `bst_ulong` and BEAM terms using the 64-bit NIF APIs**:
  ```c
  bst_ulong value = ...;

  ERL_NIF_TERM term =
      enif_make_uint64(env, (ErlNifUInt64)value);
  ```

- **Use `size_t` for allocation sizes** after validating that larger integer types fit.

- **Add checked conversions and multiplication before allocation**:
  ```c
  if (count > SIZE_MAX / element_size) {
    return exg_error(env, "Size overflow");
  }

  size_t total = (size_t)count * element_size;
  ```

- **Use matching-width loop variables**:
  ```c
  for (bst_ulong i = 0; i < len; ++i) {
    // ...
  }
  ```

#### ❌ DON'T:

- **Don't use C `unsigned long` for cross-platform 64-bit values**. It is 32-bit on 64-bit Windows and 64-bit on most 64-bit Unix systems.

- **Don't use `enif_get_ulong()` or `enif_make_ulong()` for XGBoost `bst_ulong` values**. Use the `uint64` variants.

- **Don't use `bst_ulong` for pointers**, even if pointers are also 64-bit on the current platform.

- **Don't use variable-length arrays whose size comes from XGBoost or user input**:
  ```c
  // DON'T:
  ERL_NIF_TERM terms[out_len];
  ```

  Use checked heap allocation instead:
  ```c
  if (out_len > SIZE_MAX / sizeof(ERL_NIF_TERM)) {
    return exg_error(env, "Result is too large");
  }

  ERL_NIF_TERM *terms =
      enif_alloc((size_t)out_len * sizeof(*terms));
  ```

### Input Validation

#### ✅ DO:

- **Validate all inputs before use**:

  - Check `typestr` format, for example `"<f4"`, `"<i8"`, or `"|u1"`
  - Validate shape dimensions are non-negative
  - Validate whether a particular XGBoost operation permits empty dimensions
  - Check boolean values are exactly the atoms `true` or `false`
  - Validate all lists are proper lists
  - Validate resource terms before dereferencing handles

- **Check for arithmetic overflow before every allocation or memory copy**:
  ```c
  if (count > SIZE_MAX / element_size) {
    return exg_error(env, "Size overflow");
  }

  size_t total = (size_t)count * element_size;
  ```

- **Validate that a backing binary is large enough for its declared shape and type**:
  ```c
  if (binary.size < expected_size) {
    return exg_error(env, "Binary is too small");
  }
  ```

  Exact equality may be enforced by a higher-level contiguous-array invariant, but pointer safety requires at least `binary.size >= expected_size`.

- **Validate XGBoost results before using output pointers**.

- **Validate counts fit in `size_t` before converting them for allocation or `memcpy()`**.

- **Validate type ranges with correct boolean logic**:
  ```c
  if (type < MIN_TYPE || type > MAX_TYPE) {
    return exg_error(env, "Unsupported type");
  }
  ```

#### ❌ DON'T:

- **Don't trust input data**, including values supplied by higher-level Elixir wrappers.

- **Don't skip bounds checking** before pointer arithmetic, allocation, or memory copying.

- **Don't silently default malformed booleans or metadata**. Return an error.

- **Don't allow negative dimensions or counts**.

### Resource Management

#### ✅ DO:

- **Wrap XGBoost handles in NIF resources** with cleanup callbacks.

- **Release the native resource reference after creating the BEAM resource term**:
  ```c
  void **resource =
      enif_alloc_resource(RESOURCE_TYPE, sizeof(void *));

  if (resource == NULL) {
    return exg_error(env, "Failed to allocate resource");
  }

  *resource = handle;

  ERL_NIF_TERM term =
      enif_make_resource(env, resource);

  enif_release_resource(resource);

  return exg_ok(env, term);
  ```

- **Free partially created XGBoost handles** if a later operation fails before ownership transfers into a NIF resource.

#### ❌ DON'T:

- **Don't free a handle manually after it has transferred into a resource**.

- **Don't store unowned BEAM term references in C structures**. Use an independent NIF environment and `enif_make_copy()` if a resource genuinely needs to retain a term.

## Testing

### Run tests after changes:

```bash
mix test
```

### Test your NIF changes:

1. **Safety tests** — Ensure invalid inputs are rejected without crashing the VM
2. **Boundary tests** — Cover zero lengths, maximum supported dimensions, malformed lists, and overflow attempts
3. **Lifetime tests** — Trigger garbage collection and allocation pressure around binary-backed calls
4. **Stress tests** — Repeat large allocations and native calls
5. **Sanitizer tests** — Run AddressSanitizer or Valgrind where supported
6. **Cross-platform tests** — Test Linux, macOS, and Windows where possible

Large-data tests can reveal memory issues, but they do not prove that no memory leaks exist. Prefer sanitizer-backed testing for that claim.

## Code Style

- **C code**: Follow existing formatting and use 2-space indentation
- **Elixir code**: Run `mix format` before committing
- **Comments**: Explain *why*, not *what*
- **Errors**: Return clear errors at the native boundary rather than silently correcting malformed inputs

## Pull Request Checklist

- [ ] Tests pass: `mix test`
- [ ] Code formatted: `mix format`
- [ ] No compiler warnings: `mix compile --warnings-as-errors`
- [ ] Added tests for new functionality
- [ ] Added invalid-input and boundary tests for NIF changes
- [ ] Updated CHANGELOG.md
- [ ] NIF changes validated with safety tests
- [ ] Native code tested under AddressSanitizer or Valgrind where supported
- [ ] No native addresses are exposed or reused across NIF calls
- [ ] All allocation sizes use checked arithmetic
- [ ] All temporary allocations have matching cleanup
- [ ] All XGBoost-owned outputs are copied before returning to Elixir

## Architectural Guidelines

### ArrayInterface Design

- Elixir-side Array Interface values should contain backing binaries and semantic metadata, not reusable raw addresses.
- The NIF derives addresses from `ErlNifBinary.data`.
- The NIF constructs standards-compliant pointer-bearing Array Interface JSON immediately before calling XGBoost.
- Pointer-bearing JSON is consumed before the NIF returns.
- XGBoost-returned Array Interfaces are parsed and copied inside the NIF that receives them.
- Nx tensors are reconstructed from BEAM-owned binaries and metadata.
- Do not cache Nx tensors merely to keep native addresses alive.

### DMatrix/Booster Pattern

All XGBoost interactions happen in C NIFs.

Data flow:

```text
Elixir binaries and metadata
    → C NIF
    → temporary standards-compliant Array Interface JSON
    → XGBoost C API
    → immediate copy of XGBoost-owned outputs
    → BEAM-owned binaries
```

Resources such as `DMatrix` and `Booster` use NIF resource management and cleanup callbacks.

## Documentation

When adding features:

1. Update function documentation with `@doc` in Elixir modules
2. Add examples for public API functions
3. Update `README.md` if changing user-facing behaviour
4. Add a `CHANGELOG.md` entry with a category such as Added, Changed, Fixed, or Removed
5. Document ownership and lifetime assumptions for any native pointer interaction

## Safety-Critical Areas

These areas require extra care and review:

- Any NIF that copies memory
- Array Interface JSON generation
- Pointer-to-integer conversions
- Binary size calculations
- Shape and stride validation
- XGBoost output pointer handling
- Resource cleanup
- Variable-sized result conversion
- Any conversion between `bst_ulong`, `ErlNifUInt64`, and `size_t`

## Getting Help

- **XGBoost C API changes?** Run `make check-xgboost-c-api`
- **Questions?** Open a GitHub issue or discussion

## Maintenance Tasks

### Updating XGBoost version:

1. Change `XGBOOST_GIT_REV` in `Makefile`
2. Run `make check-xgboost-c-api` to verify API compatibility
3. Review changed ownership and output-lifetime documentation in the XGBoost C API
4. Update tests if the API changed
5. Update `CHANGELOG.md`

### Adding new NIFs:

1. Declare the NIF in `c/exgboost/src/exgboost.c`
2. Implement it in the appropriate C file such as `dmatrix.c` or `booster.c`
3. Add the Elixir stub in `lib/exgboost/nif.ex`
4. Add a wrapper in the appropriate high-level module
5. Add tests in `test/nif_test.exs`
6. Add integration tests in the appropriate test file
7. Add invalid-input, overflow, and ownership tests
8. Verify that no pointer escapes the NIF invocation
